# 001 — Untrack 9.7 MB generated data dumps

- **Date:** 2026-07-15
- **Area:** Repository hygiene / Docker image size
- **Files:** `.gitignore`, `.dockerignore`, `research-cache.json`, `awesome-mcp.md`, `list.md`

## Problem

Three generated research artifacts were committed to git and copied into the
Docker image, despite never being part of any build:

| File | Size | Used by build? |
|------|------|---------------|
| `research-cache.json` | 8.38 MB | No (regenerated by `research-mcp.mjs`) |
| `awesome-mcp.md` | 1.13 MB | No (dump) |
| `list.md` | 225 KB | No (dump) |

`Dockerfile` does `COPY . .`, so all three entered the build context and the
final image. They are produced by the root `.mjs` scripts (see `Unused.md` §2/§3)
and can be regenerated on demand, so carrying them in version control only adds
~9.7 MB to every clone and image layer with zero benefit.

## Change

- Added the three files to `.gitignore` (so they are no longer tracked going forward).
- Added the three files (plus `Unused.md`, a generated report) to `.dockerignore`
  so they are excluded from the Docker build context.
- `git rm --cached` the three files — they remain on disk for local use but are
  removed from the index.

## Impact

- New clones no longer pull ~9.7 MB of regenerable data.
- Docker `COPY . .` context (and image) shrinks by the same amount; CI/image
  pushes get marginally faster and cheaper.
- Repo history still contains the old blobs; a full shrink would need
  `git filter-repo`/`BFG` (destructive, not done here).

## Risk

- **Low.** The files are not imported by any source or build step (confirmed via
  `Unused.md` §3 and a grep of the source tree). Local copies are preserved, so
  the research scripts keep working.
